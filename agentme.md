# agentme.md

## 1. 项目身份

项目名称：YaFo兼容采购助理  
当前正式版本：v3.6.14  
当前主程序文件：`YaFo兼容采购助理_v3.6.14_更新启动问候文案.py`  
项目形态：Python 单文件主生成器，Tkinter GUI，集成多个兼容采购业务子功能模块。  
开发者标识：rangerray  
GitHub：github@PLAAFR_hue

本项目服务于 YaFo / 洋帆兼容采购业务，目标是把原本分散在 Excel、手工录入、供应商文件、NC存货资料、XML导入模板和快递查询平台中的重复流程，收口到一个主生成器里，降低人工匹配、重复录入、格式错误和多文件切换成本。

## 2. 当前核心功能模块

当前主生成器包含以下功能选择：

1. 工厂直发订单  
   一次性上传多个供应商文件，自动识别供应商，批量生成采购订单 XML 和汇总表。

2. 分仓订单  
   支持北京、沈阳、南京等分仓订单文件，自动识别分仓并批量生成 XML。

3. 供应商直发表格处理  
   对供应商直发表格进行整理、转换和辅助输出。

4. 新品上新定价  
   支持兼容耗材新品按品牌定价规则生成建议价格。

5. 存货成本回填  
   选择 NC 成本表，批量导入供应商成本来源表，暂存后统一导出 Excel。

6. 销量预测与备货建议  
   导入销售订单和库存信息，生成预测销量、备货建议和 Excel 报告。

7. 快递查询平台  
   集成 Flask Web 快递单号查询平台，支持本地启动、ngrok 公网映射、数据导入、手动录入、查询、统计、导出，以及数据库回写到当前主程序。

8. NC存货CSV导入  
   支持 NC 产品信息文件导入和记住路径，供 XML 生成、匹配等流程使用。

## 7. 快递查询平台运行目录

快递查询平台优先使用以下目录：

```text
C:\Users\Administrator\Desktop\快递查询平台
```

如果该目录不存在，才回退到默认隐藏目录：

```text
C:\Users\Administrator\.yafo_purchase_assistant\express_query_platform
```

快递平台运行目录内通常包含：

```text
express_query_platform.py
express_data.db
start_express_query_platform.bat
```

关键原则：

- Web 页面导入的数据不会直接写回主生成器内嵌数据库。
- Web 页面导入、手动录入、清空、更新的数据，首先写入运行目录里的 `express_data.db`。
- `express_data.db` 是实时数据库，主程序内嵌数据库只是快照。
- v3.6.8 之后支持“回写数据库到当前主程序”，可将当前运行目录数据库重新写入当前 `.py` 文件。
- 原地回写前必须自动备份当前主程序。
- 回写时应通过临时文件和语法检查后再替换当前主程序，避免源码损坏。
- 不得在每次启动时覆盖已有 `express_data.db`。只有数据库不存在时才释放初始数据库。

## 8. ngrok 与一键启动逻辑

快递查询平台的一键启动脚本需要支持：

1. 启动 Flask 本地服务
2. 启动 ngrok 公网映射
3. 获取 ngrok Forwarding 公网链接
4. 自动复制公网链接到剪贴板
5. 自动打开公网链接
6. 打开 ngrok dashboard 方便检查状态
7. 找不到 ngrok 时仍启动本地服务，并明确提示用户

ngrok 查找路径必须包括：

```text
Windows PATH
当前快递平台目录
BAT 脚本同目录
桌面
下载目录
%LOCALAPPDATA%\ngrok
%LOCALAPPDATA%\Microsoft\WinGet\Links
```

用户曾说明 ngrok 位于：

```text
C:\Users\Administrator\AppData\Local\Microsoft\WinGet\Links
```

不要移除该路径识别。

## 12. 开发修改原则

修改主生成器时遵守以下原则：

1. 优先基于当前最新版继续修改。
2. 修改后必须更新：
   - 顶部 docstring 版本号
   - 中文窗口标题
   - 英文窗口标题
   - 维护日志
   - 必要时更新 README / agentme.md / CHANGELOG
3. 新增子功能模块时递增第二段版本号。
4. 普通打磨修复时只递增第三段版本号。
5. 保持主程序单文件可运行，除非用户明确要求拆分。
6. 不要破坏已内嵌资源。
7. 不要自动覆盖用户运行目录里的业务数据库。
8. 不要把快递查询平台的外部数据库和主程序内嵌快照混为一谈。
9. 快递数据库回写必须有备份和语法检查。
10. GUI 修复优先用布局优化、滚动区、按钮换行、可拉伸窗口，不要盲目把窗口和字体放得过大。
11. v3.5.46 的“大字号强化方案”已被用户舍弃，不应作为后续基准。

## 14. 对未来 AI / Agent 的特别说明

如果未来由 AI Agent 继续维护本项目：

- 不要猜测用户想要“拆分项目”。当前用户更偏好直接可运行的单文件主生成器。
- 不要擅自删除历史功能。
- 不要擅自改动供应商识别、NC字段匹配、XML生成逻辑，除非用户明确指出问题。
- 不要把 Web 快递平台改成必须公网才能访问。本地 `127.0.0.1:5000` 必须始终可用。
- 不要把 ngrok 作为硬依赖。没有 ngrok 时，本地快递查询平台仍应能启动。
- 不要在没有备份的情况下修改当前主程序源码。
- 不要把导入后的快递数据自动写回公开仓库。
- 遇到 UI 按钮显示不全，优先使用可滚动区域、多行网格、可拉伸窗口、按钮换行解决。
- 遇到用户要求“打包”，优先提供运行目录 zip，包含主程序、requirements、启动 bat、安装依赖 bat、README、CHANGELOG、agentme.md 和 .gitignore。
- 所有最终交付物必须给出可下载链接。
